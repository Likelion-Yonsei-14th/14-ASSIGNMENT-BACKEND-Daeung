# 📌 결제 시스템 리팩토링 과제

## 🎯 과제 개요

기존의 결제 시스템 코드를 분석하고,
객체지향적으로 구조를 개선하는 것을 목표로 합니다

---

## 1️⃣ 기존 코드의 문제점

- 캡슐화 X: `Payment` 클래스의 `type`, `amount`가 `public`이라 외부에서 마음대로 바꿀 수 있어, 결제 정보의 일관성을 객체 안에서 보장하기 어렵다.
- 책임 분리 문제: 금액 검증(`amount <= 0`)과 결제 수단 분기(`card`, `kakao` 등)가 모두 `PaymentService`에 있어, `Payment`는 데이터만 담는 역할에 그쳤다. 결제가 가지는 속성에 대한 검증을 다른 클래스에서 하는 것이 문제이다.
- Service 역할 과다: `PaymentService`가 “결제 요청”뿐 아니라 `Payment` 생성, 검증, 출력 모든 역할을 수행한다. 기능 확장시 이렇게 역할이 과다해지면 Service가 비대해지고, 다른 service에서 `Payment`클래스를 이용한다면 또 같은 코드를 작성해야한다. 만약 그렇게 하지않고 결제로직을 만들면 검증없이 쓰는 코드가 된다.

---

## 2️⃣ 로직 이동 내용

### 🔹 변경 전

- `PaymentService.pay()` 안에서 `Payment` 객체를 만든 뒤 필드를 직접 대입하고, 금액 검증과 결제 수단별 출력까지 한 메서드에서 처리했다.

### 🔹 변경 후

- 금액 검증과 `card` / `kakao` / 그 외 분기 로직을 `Payment.process()`로 옮겼고, 결제 수단 확인은 `isCard()`, `isKakao()`에서 담당한다.
- `PaymentService`는 `new Payment(type, amount)` 이후(생성) `payment.process()` 호출만 한다.

---

## 3️⃣ 구조 개선 내용

- 필드 캡슐화: `type`, `amount`를 `private final`로 두고 생성자에서만 초기화해, 외부에서 필드를 직접 수정할 수 없게 했다.
- 책임 분리: `Payment`가 `process()`로 실제 결제 흐름(검증·분기·출력)을 수행하고, 타입 판별 메서드로 자기 상태를 해석하도록 했다. 즉 객체 정보에 관한 처리 메소드를 생성했다.
- Service 단순화: `PaymentService`는 “결제를 요청한다”는 역할만 하도록, 도메인 로직은 `Payment`에 두었다.
